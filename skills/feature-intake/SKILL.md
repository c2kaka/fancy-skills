# Feature Intake —— 从原型 + 接口文档反推单一真相源

## 这个 skill 解决什么

产品现在的交付形态是 **HTML 原型 + 后端接口文档**，没有完整 PRD，没有需求澄清过程。
这两份都是**不完整的 spec**，它们之间有一道天然**缝隙**——隐式业务行为：

- 状态流转（用户在不同入口、不同操作下表单怎样流转）
- 字段依赖与动态候选
- 边缘 case（孤儿引用、失效清理）
- 新建态 vs 编辑态差异
- 错误处理、展示值 vs 提交值

**Bug 基本都住在缝隙里。** AI 特别擅长用"看起来很像真的"的默认值填缝隙——它不问，它假设，
而且假设出来的代码 review 时很难发现。本 skill 把缝隙显式化、逼成人工决策，
让人做决策、AI 做执行。

**真实数据**：本仓库 cloud-files-to-cogdb 功能开发中事后补救的 7 条语义，对照原始三份材料，
0 条有显式规定（3 条完全空白、4 条只能反推）。详见 `references/gap-patterns.md`。

## 什么时候用

- 用户说"我拿到原型和接口文档了，帮我理一下这个功能"
- 用户说"产品没给 PRD，只有原型和接口"
- 用户说"原型和接口对不上 / 有矛盾 / 看不出某字段怎么处理"
- 用户要开始一个新前端功能，手上有原型 HTML + 接口文档
- 用户提到"需求提取 / 反推需求 / feature intake / spec 撰写"

## 核心纪律（先读）

1. **不允许用"合理默认值"填缝隙**。每处缝隙要么从材料反推结论（标置信度），要么标 `TODO:` 等人工裁决。
2. **每条结论必须标置信度**：`[产品确认]` / `[接口确认]` / `[原型确认]` / `[实现现状]` / `[协议TODO]`。
3. **主动用 5 类缝隙模式扫描**，不要等缝隙自己冒出来（见下"缝隙扫描"）。
4. **协议级未定**（枚举白名单、命名不一致、字段是否存在）单独收一节，不混进业务缝隙。
5. **产出是给人看的**：发现的每个缝隙、每个决策、每个 TODO 都要落在文档里，不能只藏在脑子里。

## 工作流

### Step 1：拿到三份材料，先建 spec 文件

复制本 skill 内置模板到功能目录：

```bash
cp .agents/skills/feature-intake/assets/feature-intake-spec.md \
   .agent-local/docs/prds/<功能名>/<功能名>-intake-spec.md
```

模板内有逐节填写指引和置信度标注约定，照着填。产出格式骨架见下文"产出格式速览"。

如果三份材料不齐全，先问用户要。常见情况：
- 只有原型没接口文档 → 先做能做的部分，接口相关字段全标 `[协议TODO]`
- 只有接口文档没原型 → 缺视觉契约，标 `[协议TODO]` 等原型
- 有产品逻辑文档（哪怕是一段聊天记录）→ 当作"产品确认"来源

### Step 2：分别读三份材料，填"材料分工表"

在 spec 第 2 节填三列：每份材料**覆盖什么、不覆盖什么**。

**这一步本身就能暴露缝隙**。当你发现"产品逻辑讲了 X，但接口文档没对应字段"或
"原型画了 Y，但接口 JSON 里是 null"——那就是缝隙。

### Step 3：交叉对照，用 5 类缝隙模式主动扫描

这是整个 skill 的核心。逐类问，不要跳过。详见下文"缝隙扫描"，需要逐条对照时读
`assets/gap-scan-checklist.md`，判断归类时读 `references/gap-patterns.md`。
每发现一处，在 spec 第 7 节"缝隙清单"记一行（编号 / 描述 / 模式类型 / 裁决或TODO / 置信度）。

### Step 4：每条缝隙逼成裁决或 TODO

- 能从某份材料反推的：标置信度，写结论 + 依据。
- 无法反推的：标 `TODO:`，写清"找谁确认什么"。TODO 要具体可搜索，例如
  `TODO: 找后端确认 timeFunction 内置项是否为最终白名单（联调指南建议 5 个）`，
  而不是 `TODO: 确认枚举`。

### Step 5：产出"字段×状态矩阵"和"依赖主线"

这是给编码阶段的第一手输入，不是可选的装饰：

- **字段×状态矩阵**（spec 第 6 节）：行=字段，列=关键模式/状态，单元格填"显示/隐藏→归一化值"。
- **依赖主线**（spec 第 5 节）：至少三条——① 总开关字段驱动显隐归一化（含新建/编辑态差异）；
  ② 异步触发的失效逻辑；③ 保存校验链（前端即时 vs 后端最终）。

### Step 6：产出"视觉契约"，特别区分显示值与提交值

在 spec 第 8 节记：布局结构、关键默认态、文案清单、五态完整性。
**最关键**：原型上"显示了一个值"但接口里"对应字段是 null"的，必须标出来——
这是最隐蔽的缝隙（模式 D）。

### Step 7：交付给人工裁决

把 spec 交给用户，逐条过缝隙清单和 TODO。这一步是"人做决策"的核心。
AI 产出的草稿 + 人工裁决 = 最终的单一真相源。

## 产出格式速览

完整模板在 `assets/feature-intake-spec.md`（有逐节填写指引）。这里内联精简骨架，
让你无需翻文件就知道产出长什么样。**置信度标注是核心工具**，每条结论必须带：

- `[产品确认]` 产品逻辑/PRD 原文 · `[接口确认]` 后端接口文档 · `[原型确认]` HTML 原型
- `[实现现状]` 当前代码已落地 · `[协议TODO]` 待澄清，未当既定事实

```markdown
# <功能名> Feature Intake Spec

## 1. 业务背景
### 1.2 范围边界（做什么 / 不做什么 / 不暴露给前端）

## 2. 材料分工（三份输入各覆盖什么）   ← 填表本身就能暴露缝隙
| 材料 | 路径 | 覆盖 | 不覆盖 |

## 3. 核心业务规则
| 前端文案 | 枚举 | 运行逻辑 | 写入逻辑 |  ← 标置信度

## 4. 配置字段详解（按步骤/区块）
| 原型字段 | 后端字段 | 必填 | 说明 | 置信度 |

## 5. 前后依赖关系   ← 至少三条主线
### 5.1 总开关字段驱动显隐归一化（含新建/编辑态差异表 + 归一化规则）
### 5.2 异步触发的失效逻辑
### 5.3 保存校验链（前端即时 vs 后端最终）

## 6. 字段×状态矩阵   ← 给编码的第一手输入
| 字段 | <状态1> | <状态2> | 单元格填"显示/隐藏→归一化值" |

## 7. 缝隙清单   ← 用 5 类模式扫出来的，逐条标模式类型
| # | 缝隙 | 模式(A-E) | 裁决/TODO | 置信度 |

## 8. 视觉契约   ← 特别区分"显示值"和"提交值"
布局结构 / 关键默认态 / 文案清单 / 五态完整性

## 9. 协议 TODO（契约级未定，单独成节，不混进第7节）
| 项目 | 当前结论 | 风险/后续动作 |

## 10. 接口与文件归属（复用既有，不建第二套）

## 12. 完成定义（验收清单来源）
```

**填好的范例**：`.agent-local/docs/prds/云存储中文件导入/file-load-requirement-overview.md`
（cloud-files-to-cogdb 的 Intake Spec，含置信度标注和 5 类缝隙的实际应用）。

## 缝隙扫描（5 类模式）

每一类都给"信号 → 必问 → 为什么危险"。真实案例对照见 `references/gap-patterns.md`。

### 模式 A：入口差异（新建态 vs 编辑态）
- **信号**：字段有"默认值 / 自动行为 / 自动填充"。
- **必问**：编辑态回填的已保存值，会不会被这个默认行为覆盖？自动行为的触发时机是什么（进页面？切换？用户操作？）？
- **为什么危险**：AI 看到"增量模式默认开启"会写成"只要进入增量模式就开启"，把编辑态已保存的关闭状态也覆盖。

### 模式 B：字段依赖与动态候选
- **信号**：一个字段的候选来自"另一个接口返回 / 另一组用户填写的数据 / 另一个字段的值"。
- **必问**：候选在哪一刻刷新？多来源合并的去重规则和优先级？候选为空时怎么办？编辑态旧值要不要兜底保留？
- **为什么危险**：AI 会写成"候选写死"或"只取一个来源"，漏掉合并去重和兜底。

### 模式 C：孤儿引用清理（被引用项变化导致引用失效）
- **信号**：存在"引用关系"（partition 引用某字段、mapping 引用某列、外键引用某选项）。
- **必问**：被引用项被删/改名/禁用后，引用方要不要清空？要不要提示？会不会静默提交失效值？
- **为什么危险**：AI 不会主动处理引用失效，会留下指向已删字段的悬空引用，提交时变成脏数据。

### 模式 D：展示值 ≠ 提交值（UI 显示但不落库）
- **信号**：原型上"显示了一个值"，但接口里对应字段"可能是 null / 没有对应字段"。
- **必问**：这个显示值要不要提交？提交到哪？还是纯展示？JSON 示例里是 null 是"恰好这个例子"还是"永远"？
- **为什么危险**：AI 看到原型显示了值，会自然写进提交字段。只有对照 JSON 示例的 null 才能发现。**这类最隐蔽。**

### 模式 E：会话草稿与切换时的保留/清理
- **信号**：有"开关 / 模式切换 / 步骤跳转 / Tab 切换"。
- **必问**：关闭再开启草稿还在吗？切到互斥模式清不清？切回来恢复到什么？原型文案自相矛盾时以谁为准？
- **为什么危险**：AI 会按"最简单"做——要么全清要么全留，不区分"互斥切换清、同模式开关留"。原型常自带矛盾文案。

### 横切：契约级未定（单独收一节，不混进上面）
枚举白名单、命名不一致（如同一概念 AZURE_BLOB vs ADLS）、字段是否存在（原型有合同无）、
相对路径 vs 绝对路径。这类不是"表单状态机"问题，是"契约没谈拢"，单独放 spec 第 9 节。

## 决策边界：哪些必须人定，哪些可交 AI

| 必须人定（AI 只能提示） | 可放心交给 AI（人 review） |
|---|---|
| 歧义与取舍（原型矛盾时以谁为准） | 样板代码、字段注册、i18n 词条 |
| 协议假设（枚举白名单、字段存在性） | 字段映射（如 validate/save 双结构转换） |
| 架构边界（路由、API 归属、全局状态） | UI 还原（按视觉契约 + token 映射） |
| 产品假设（任何"AI 觉得合理就写"的业务规则） | 从 spec 派生测试用例 |
| 安全边界（表达式要不要前端求值） | 类型定义、barrel export |

重要决策记进 `.agent-local/docs/playbook/templates/decision-log.md`（决策/理由/依据/影响范围）。

## 完成标志

Intake 完成意味着：
- [ ] spec 的材料分工表、字段×状态矩阵、依赖主线都填了
- [ ] 5 类缝隙模式逐类扫过，每处缝隙记进缝隙清单
- [ ] 每条缝隙要么裁决（标置信度+依据）要么 TODO（具体可搜索）
- [ ] 视觉契约区分了"显示值"和"提交值"
- [ ] 协议 TODO 单独成节
- [ ] 用户已逐条过完缝隙清单

此时产出的 spec 就是后续设计、编码、测试的单一真相源。
