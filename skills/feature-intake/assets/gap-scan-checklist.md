# 缝隙扫描 Checklist（5 类模式）

> 用法：每做一次 Feature Intake，拿这 5 类**主动去问**，而不是等缝隙自己冒出来。  
> 每发现一处，回填到 `feature-intake-spec.md` 的第 7 节"缝隙清单"。  
> 真实案例：cloud-files-to-cogdb 的 7 条事后补救语义，0 条在原始材料里有显式规定——这 5 类就是从那 7 条归纳出来的。

## 共同根源（先理解再扫描）

产品逻辑文档只写"跑起来怎样"、接口文档只写"提交什么字段"、原型只画"长什么样"。**三者之间天然缺一层："用户在不同状态和操作下，表单状态机怎样流转"**。这层就是缝隙。

---

## 模式 A：入口差异（新建态 vs 编辑态）

**信号**：字段有"默认值 / 自动行为 / 自动填充"。

**必问**：
- [ ] 编辑态回填的已保存值，会不会被这个"默认行为"覆盖？
- [ ] 自动启用 / 自动填充的**触发时机**是什么？（进入页面？切换模式？用户操作？）
- [ ] 新建态和编辑态的行为是否一致？不一致的话各自是什么？

**案例**：cloud-files 第 1、2 条——新建态 Full→增量自动启用 Target config；编辑态必须尊重 `targetConfigEnabled=false` 不强制开启。

**高频 bug**：AI 看到"增量模式默认开启"会写成"只要进入增量模式就开启"，把编辑态已保存的关闭状态也覆盖了。

---

## 模式 B：字段依赖与动态候选

**信号**：一个字段的候选来自"另一个接口返回 / 另一组用户填写的数据 / 另一个字段的值"。

**必问**：
- [ ] 候选列表在哪一刻刷新？（接口返回时？字段变化时？进入步骤时？）
- [ ] 多个来源合并时，去重规则是什么？优先级呢？
- [ ] 候选为空时怎么显示？是否允许手输？
- [ ] 编辑态已保存但当前候选里没有的值，要不要兜底保留？

**案例**：cloud-files 第 3 条——Partition field 候选动态合并 preview columns + 启用的 audit fields，按名去重，编辑态旧值兜底。

**高频 bug**：AI 会写成"候选写死"或"候选只取一个来源"，漏掉合并去重和兜底。

---

## 模式 C：孤儿引用清理（被引用项变化导致引用失效）

**信号**：存在"引用关系"——partition 引用某字段、mapping 引用某列、外键引用某选项。

**必问**：
- [ ] 被引用项被删 / 改名 / 禁用后，引用方要不要清空？
- [ ] 清空时要不要提示用户？还是静默清空？
- [ ] 会不会静默提交一个失效的引用值？
- [ ] 清空后是否触发必填校验提示？

**案例**：cloud-files 第 4 条——删 audit field 后，partitionColumn 若引用了它就清空并交必填校验。

**高频 bug**：AI 不会主动处理"引用失效"，会留下指向已删字段的悬空引用，提交时变成脏数据。

---

## 模式 D：展示值 ≠ 提交值（UI 显示但不落库）

**信号**：原型上"显示了一个值"，但接口里对应字段"可能是 null / 没有对应字段"。

**必问**：
- [ ] 这个显示值要不要提交？
- [ ] 如果提交，提交到哪个字段？
- [ ] 如果不提交，它是纯展示吗？展示规则是什么？
- [ ] 接口 JSON 示例里这个字段是 null，是"恰好这个例子是 null"还是"永远是 null"？

**案例**：cloud-files 第 6 条——`DELETE_MATCHING_PARTITION` 展示 `schedule.date()` 但 `cleanupCondition` 提交 null。**这类最隐蔽**——看原型以为要存，看 JSON 才发现是 null。

**高频 bug**：AI 看到原型显示了值，会自然地把它写进提交字段。只有对照 JSON 示例的 null 才能发现。

---

## 模式 E：会话草稿与切换时的保留/清理

**信号**：有"开关 / 模式切换 / 步骤跳转 / Tab 切换"。

**必问**：
- [ ] 关闭再开启某个开关，之前的草稿还在吗？
- [ ] 切到互斥的模式，当前模式的草稿清不清？
- [ ] 切回来时，恢复到什么状态？
- [ ] 原型文案自相矛盾时（常见！），以谁为准？

**案例**：cloud-files 第 5 条——增量模式内关闭再开启保留草稿；切 Full 清空。原型还自带矛盾文案（"clear all" vs "resets unedited"）。

**高频 bug**：AI 会按"最简单"的实现做——要么全清要么全留，不会区分"互斥切换清、同模式开关留"。

---

## 横切：契约级未定（非业务行为，而是协议本身没定）

**信号**：枚举白名单、命名不一致、字段是否存在、相对路径 vs 绝对路径、原型有字段但合同没有。

**必问**：
- [ ] 这套枚举值是完整白名单，还是示例？
- [ ] 同一概念在不同地方命名不一致（如 AZURE_BLOB vs ADLS），以谁为准？
- [ ] 原型有的字段，接口合同里没有，要提交吗？
- [ ] 接口示例的路径是相对还是绝对？前端要不要自己拼协议头？

**处理方式**：这类**单独收一节**（spec 第 9 节"协议 TODO"），不混进业务缝隙。因为它们不是"表单状态机"问题，而是"契约没谈拢"。

---

## 扫描纪律

1. **主动扫，不是被动等**：5 类模式逐类问一遍，哪怕你觉得"这个功能应该没有"。
2. **每条缝隙要么裁决，要么 TODO**：不允许"AI 觉得合理就默认"。
3. **裁决要留依据**：写明结论 + 依据来自哪份材料 / 哪次确认。
4. **TODO 要具体可搜索**：`TODO: 找后端确认 timeFunction 内置项是否为最终白名单（当前联调指南建议 5 个）`，而不是 `TODO: 确认枚举`。
5. **横切契约问题不混进业务缝隙**：保持两节分离，清晰度更高。
