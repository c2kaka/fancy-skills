# 5 类缝隙模式：真实案例对照

> 本文件是 feature-intake skill 的参考文档。当你在扫描缝隙、判断"这条算哪一类"时读它。  
> 所有案例来自本仓库 `cloud-files-to-cogdb` 功能（Cloud Files to CogDB 文件加载任务）。  
> 硬数据：开发中事后补救的 7 条语义，对照原始三份材料（产品逻辑 / 接口文档 / 原型 HTML），**0 条有显式可照做的规定**——3 条完全空白，4 条只能反推。

## 案例背景

三份原始材料各自的角色：
- **产品逻辑** (`文件加载产品逻辑.md`)：只讲运行时/写入的后端数据行为，不提前端表单状态。
- **接口文档** (`2026-07-14-cloud-files-to-cogdb-frontend-interaction.md`)：字段口径 + JSON 契约，不解释字段联动。
- **原型** (`Cloud_Files_to_CogDB_原型_V3.html`)：UI 形态 + 静态文案 + 默认态，无显式状态机，且自带矛盾文案。

三者之间天然缺一层："用户在不同状态和操作下，表单状态机怎样流转"。这层就是缝隙。

## 模式 A：入口差异（新建态 vs 编辑态）

### 案例 A1：新建态切换应自动启用配置
- **事后补的语义**：新建态从 Full load 切到增量模式时，自动启用 Target config，默认填 `_ingestion_date + DELETE_MATCHING_PARTITION`。
- **三份材料怎么写的**：
  - 产品逻辑：无。
  - 接口文档：只说"增量模式出现 Target config"，不提自动启用。
  - 原型：有默认勾选的**视觉**，但**不区分新建/编辑态，不规定"切换触发自动启用"**。
- **缝隙判定**：触发时机 + 新建/编辑区分完全没写。AI 若不问，会写成"进入增量模式就启用"，连编辑态已关闭的也覆盖。
- **正确处理**：标 `[协议TODO]`，找产品确认"自动启用的触发时机"和"编辑态是否豁免"，结论：新建态切换才触发，编辑态尊重后端返回值。

### 案例 A2：编辑态必须尊重已保存的关闭状态
- **事后补的语义**：编辑态若后端返回增量模式但 `targetConfigEnabled=false`，必须尊重，不强制开启。
- **三份材料怎么写的**：三份**完全空白**——产品逻辑无、接口文档无、原型只有个 "Editing" 文案但不规定行为。
- **缝隙判定**：典型的新建/编辑态差异，三份材料零提及。
- **正确处理**：标 `[协议TODO]`，这是最典型"只会出现在 bug 里"的隐式行为。

**扫描信号**：只要字段有"默认值/自动行为"，立刻问——编辑态回填值会不会被覆盖？

## 模式 B：字段依赖与动态候选

### 案例 B1：Partition field 候选动态合并多来源
- **事后补的语义**：Partition field 改为下拉，候选动态合并 preview columns + 当前启用的 audit fields + 编辑态旧值兜底，按字段名去重。
- **三份材料怎么写的**：
  - 产品逻辑：无。
  - 接口文档：只说"`targetConfigEnabled=true` 时 partitionColumn 必填"，没说候选来源。
  - 原型：有句 "Partition key can use source fields and audit fields"，但没说动态、没说去重、没说编辑态兜底。
- **缝隙判定**：字段依赖与候选来源完全没写。AI 会写成"候选写死"或"只取一个来源"。
- **正确处理**：标 `[协议TODO]` 确认候选合并规则，结论回填。

**扫描信号**：一个字段的候选来自"另一个接口返回 / 另一组用户填写的数据"，就问——候选何时刷新？去重？上游没了？

## 模式 C：孤儿引用清理

### 案例 C1：删 audit field 后清空失效的 partitionColumn
- **事后补的语义**：用户删除/禁用/改名已选 audit field 后，若该字段不再属于有效候选，清空 partitionColumn 并交必填校验，不静默切到其他字段。
- **三份材料怎么写的**：三份**完全空白**——无任何提及"引用失效后怎么办"。
- **缝隙判定**：典型"上游字段变化→下游引用失效"边缘 case。AI 不会主动处理，会留下悬空引用，提交脏数据。
- **正确处理**：标 `[协议TODO]` 确认清空策略和提示方式。

**扫描信号**：凡有"引用关系"（partition 引用某字段、mapping 引用某列），就问——被引用项删/改名后引用方怎么办？

## 模式 D：展示值 ≠ 提交值（最隐蔽）

### 案例 D1：schedule.date() 展示但 cleanupCondition 提交 null
- **事后补的语义**：`DELETE_MATCHING_PARTITION` 模式固定展示只读 `schedule.date()` 与 `${schedule_date}`，**不写入** cleanupCondition，validate/save 继续提交 `cleanupCondition=null`。
- **三份材料怎么写的**：
  - 产品逻辑：无。
  - 接口文档：6.3 节 validate 示例里 `cleanupCondition=null`（可**反推**），但**没明说**"这个值是内置展示不落库"。
  - 原型：**展示了** `schedule.date()`，但**不告诉**你它要不要提交。
- **缝隙判定**：展示值 vs 提交值的差异是隐式的，只能靠看 JSON 示例反推。**这类最隐蔽**——看原型以为要存，看 JSON 才发现是 null。
- **正确处理**：标 `[接口确认]`（从 JSON 示例反推）+ 说明"展示不落库"。issue 第 22 行点破"原型中的 schedule.date() 是内置策略展示，不是自定义条件字段"。

**扫描信号**：原型上"显示了值"但接口里"对应字段可能是 null"，必须标出来。问——这个值提交吗？提交到哪？还是纯展示？

## 模式 E：会话草稿与切换时的保留/清理

### 案例 E1：关闭再开启保留草稿，切 Full 清空
- **事后补的语义**：增量模式内关闭再开启 Target config 时保留当前会话草稿；切换到 Full load 时清空草稿。
- **三份材料怎么写的**：三份对草稿策略零提及。更糟的是**原型自带矛盾文案**：
  - 一句 "Switching the build mode will clear all current configuration on this task"（暗示切换就清空）
  - 一句 "resets unedited targets to the new rule"（暗示保留已编辑）
  - 两句话打架。
- **缝隙判定**：草稿/会话状态流转没写，且原型自带歧义。AI 会按"最简单"做——要么全清要么全留，不区分"互斥切换清、同模式开关留"。
- **正确处理**：标 `[协议TODO]`，把原型矛盾文案点出来让人工裁决。

**扫描信号**：凡有"开关 / 模式切换 / 步骤跳转"，就问——切回来草稿在吗？切互斥模式清吗？原型矛盾时以谁为准？

## 横切：契约级未定（非业务行为，是协议没定）

这类**单独收一节**（spec 第 9 节），不混进上面 5 类。cloud-files 案例：

| 项目 | 当前结论 | 问题 |
|---|---|---|
| timeFunction 内置项个数 | 联调指南建议 5 个 builtinName | 后端未确认是否最终白名单 |
| derive-target 接口 | P0 不调用，前端本地推导 | 合同未定 |
| browser path 形态 | 示例有相对路径 vs 完整 URI | 前端不自己拼协议头，等后端返回 canonical path |
| ADLS 命名 | Connectors 用 AZURE_BLOB，文件接口用 ADLS | 命名不一致，仅筛选展示用 |
| audit fieldType 白名单 | 原型 6 种类型，指南未给完整白名单 | 等后端确认 |
| 文件并发读取数 | 原型有字段，保存合同无字段 | 等后端模型明确 |

**区分依据**：这些不是"表单状态机怎样流转"，是"契约本身没谈拢"。分开收口，清晰度更高。

## 扫描纪律小结

1. **主动扫，不是被动等**：5 类逐类问，哪怕觉得"这个功能应该没有"。
2. **每条缝隙要么裁决要么 TODO**：不允许"AI 觉得合理就默认"。
3. **裁决留依据**：结论 + 来自哪份材料 / 哪次确认。
4. **TODO 要具体可搜索**：`TODO: 找后端确认 X 的 Y（当前材料只说 Z）`。
5. **横切契约问题不混进业务缝隙**：保持 spec 第 7 节（业务缝隙）和第 9 节（协议 TODO）分离。
