# Interview Prep From Project

## Overview

针对一个已有的代码项目，产出三份相互衔接的中文 Markdown 文档，形成一条「提炼 → 考察 → 回答」的面试准备链：

1. **项目亮点 `highlights.md`** — 从代码与文档中提炼可写进简历的亮点（架构设计、技术难点、性能优化、设计模式等）
2. **面试问题 `questions.md`** — 站在面试官角度，针对亮点追问、设坑、深挖，落地成结构化问题清单
3. **参考答案 `answers.md`** — 针对每个问题给出可直接背诵/复述的参考答案

三份文档顺序产出：亮点是问题的素材，问题是答案的索引。不要跳步——没有亮点文档就直接生成问题，问题会失去依据、泛泛而谈。

## Input

开始前确认：

- **目标项目**：仓库根目录路径，或某个模块/子系统路径。用户未给定时主动询问。
- **作者角色与范围**：用户在项目里负责哪些模块、处于什么角色（owner / 核心开发 / 参与者）。这决定了哪些亮点可以「据为己有」地写进简历。用户说不清时，提示其补充，但不要因此卡住整个流程。
- **目标岗位与层级**：社招/校招、前端/后端/全栈、目标职级。影响问题的深度与切入点。用户未说明时按「中高级后端/全栈社招」的默认深度推进，并在文档头部标注假设。

将这三项写进每份文档头部的「元信息」区，便于后续迭代时对齐。

## Workflow

### 阶段 0 · 读懂项目再动笔

不要凭 README 和目录名臆测亮点。先用最少的代价建立对项目的准确认知：

- 读 README、架构图、package/依赖清单、核心入口文件
- 用 `git log --oneline -n 30`、`git log --stat` 看近期改动密度，定位活跃模块
- 对每个候选亮点，回到代码里验证：找到对应的实现、配置、测试，确认它真实存在、真实生效、真实是「亮点」而非平淡实现

**诚实原则**：只输出能在代码里找到证据的亮点。如果某个常见亮点（如「微服务」「高并发」）在代码里没有实质支撑，不要为了凑数而写。宁可亮点少而实，不要多而虚——面试官追问时虚的亮点会直接崩盘。

判断一个候选是否够格成为亮点，用这个三问自检（三项全过才保留）：

1. **有证据** — 能在代码/配置/文档中指出具体位置吗？
2. **有难度** — 这件事是否超出了「按文档照做」的范畴？是通用脚手架自带，还是团队真正设计/取舍过的？
3. **能讲清楚为什么** — 为什么这么做、权衡了什么、放弃了什么？

### 阶段 1 · 提炼项目亮点 → `highlights.md`

按以下类别系统性地扫描项目，每发现一个候选亮点，先用阶段 0 的三问自检，通过后再落笔：

- **架构设计**：分层结构、模块划分、核心抽象、边界划分、演进历史
- **技术难点**：复杂业务逻辑、并发/一致性、状态管理、异常处理、兼容性、数据迁移
- **性能优化**：关键路径的优化点、指标（延迟/QPS/内存/体积）的前后对比、瓶颈定位过程
- **设计模式**：真正解决了问题的模式应用（策略/工厂/责任链/观察者等），说明它解决了什么
- **工程实践**：测试策略、CI/CD、可观测性、错误恢复、发布与回滚机制
- **其他**：安全、可扩展性、跨团队协作中体现技术判断的地方

对每个亮点，落到文档时必须包含（完整结构见 [references/highlights-template.md](references/highlights-template.md)）：

- 一句话亮点（可直接粘贴进简历的那句）
- 技术背景：为什么需要解决这个问题
- 实现要点：怎么做的，关键代码/文件位置
- 难点与权衡：卡在哪、放弃了什么、为什么这么选
- 可量化结果：有数据写数据，没有就写定性收益，并标注「待补数据」

用 [references/highlights-template.md](references/highlights-template.md) 作为输出骨架。完成阶段 1 后再进入阶段 2。

### 阶段 2 · 从面试官视角挖掘问题 → `questions.md`

拿到 `highlights.md` 后，切换到面试官心态：目标是验证候选人是否真的懂、是否只是照抄、以及能被逼到多深。对每个亮点按问题类型逐一挖：

- **追问型（验证深度）**：为什么这么设计？备选方案是什么？为什么没选它们？
- **边界型（查盲区）**：并发 / 极端输入 / 故障 / 数据量翻 10 倍时会怎样？
- **陷阱型（查真伪）**：这是通用框架自带的，还是你真正设计的？改动前是什么样？
- **延伸型（查迁移能力）**：换一种技术栈/规模，你会怎么做？
- **基础关联型**：这个亮点背后涉及哪些基础原理？把它们变成显式问题考一遍。

每个问题都要标注「针对哪个亮点」和「考察点」，方便阶段 3 对应作答，也方便用户后续选择性准备。用 [references/questions-template.md](references/questions-template.md) 作为输出骨架。

覆盖度自检：每个亮点至少配 1 个追问型 + 1 个边界型 + 1 个陷阱型问题。如果某个亮点配不出三类问题，说明阶段 1 写得不够深，回到阶段 1 补充。

### 阶段 3 · 生成参考答案 → `answers.md`

针对 `questions.md` 里的每个问题作答。答案要可以直接用于面试现场复述，不是写给自己看的草稿：

- **先给结论，再展开**：一句话直接回答，再补充理由与细节
- **紧扣亮点，不空谈八股**：答案必须回挂到本项目的具体实现，而不是抄通用博客。每条答案至少包含一处「在我们这个项目里……」的具体细节
- **承认局限**：对不确定或有争议的点，明确写「这一点我当时没有深入/存在替代做法」，不要硬编一个无懈可击的假答案——面试官更欣赏诚实+思考
- **标注深度**：每个答案标注 `[核心]` / `[进阶]` / `[加分]` 三档，帮助用户分配复习时间

用 [references/answers-template.md](references/answers-template.md) 作为输出骨架。

## Output Contract

三份文档默认放在项目根目录下的 `interview-prep/` 文件夹（用户指定其他位置时从其指定）：

```
<interview-prep>/
├── highlights.md     # 阶段 1 产出
├── questions.md      # 阶段 2 产出,每题回挂亮点编号
└── answers.md        # 阶段 3 产出,每题回挂问题编号
```

三份文档用统一的亮点编号（如 `H1`、`H2`）互相引用：`questions.md` 里每题标注「针对 Hx」，`answers.md` 里每题标注「对应 Qx / Hx」。这样用户后续增删一个亮点时，能快速定位需要同步更新的问题和答案。

每份文档头部写明：目标项目、用户角色与范围、目标岗位假设、生成日期、基于的代码版本（`git rev-parse --short HEAD`）。

## Anti-Patterns

避免这些常见错误：

- **不读代码就写亮点** — 仅凭 README 或目录名臆测，产出经不起追问的虚亮点
- **把通用框架能力当成自己的亮点** — 「用了 Spring Boot」「用了 React」不是亮点，解决了一个真问题才是
- **只写机制不写权衡** — 只说「用了 XX 模式」，不说为什么用、放弃了什么、不这么做会怎样
- **跳步生产** — 没有亮点就生成问题，问题会脱离项目泛泛而谈
- **问题全是「为什么这么设计」** — 缺少边界型、陷阱型问题，覆盖不了面试官的真实考察维度
- **答案照抄八股文** — 答案和项目脱节，面试官一追问「你具体是怎么做的」就露馅
- **编造量化数据** — 没有真实指标就写「性能提升 50%」。没有数据就标「待补数据」，绝不编造
- **贪多** — 塞满二十个平庸亮点，不如五个扎实、能扛三轮追问的亮点
